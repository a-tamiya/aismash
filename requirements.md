# プロンプトファイターズ 要件定義書

## 1. プロジェクト概要

### 1.1 タイトル

**プロンプトファイターズ**

### 1.2 概要

本プロジェクトは、Unityで開発する**1vs1ローカル対戦型の2D格闘ゲーム**である。

プレイヤーは試合前に、キャラクター名と特徴を自由入力する。AIはその入力をもとに、キャラクターの見た目、4スロット分の技（基本技A/B/C・スマッシュ）＋掴み/投げパラメーター、技名、技の構造、技パラメータを生成する。生成されたキャラクター同士が、地続きの2DステージでHPを削り合う。

本作の中心体験は、**「自分の考えたキャラクターがAIによってゲーム内ファイターとして形になり、その場で対戦できること」**である。

### 1.3 想定利用シーン

本作は、主に**オープンキャンパスの出し物**として展示することを想定する。

想定プレイヤーは高校生や来場者であり、短時間で直感的に遊べることを重視する。

### 1.4 開発エンジン

- Unity
- 2Dプロジェクト
- C#

## 2. ゲームコンセプト

### 2.1 コンセプト文

> 言葉からファイターを生み出せ。  
> 名前と特徴を入力すると、AIが見た目と技を生成。  
> 世界に一体だけのキャラクターで、友達と戦う2D格闘ゲーム。

### 2.2 企画の核

本作では、プレイヤーがキャラクターを直接デザインするのではなく、**キャラクター名と特徴をAIに渡し、AIがそれを解釈してファイター化する**。

例：

```text
キャラクター名：シャドウキャット
特徴：黒い炎をまとった素早い猫の剣士。近距離で連続攻撃し、相手を怯ませる戦い方が得意。
```

AI生成例：

```text
見た目：黒い炎をまとった猫耳の剣士。細身で素早そうな2Dアニメ立ち絵。
基本技A（右ボタン）：黒炎みだれ斬り
基本技B（下ボタン）：影火球
基本技C（左ボタン）：闇猫ステップ
スマッシュ（はじき＋右）：ナイトメアラッシュ
掴み：（パラメーターのみ生成・技名なし）
```

このように、プレイヤーが考えるのは**キャラクター名と特徴**であり、AIがそこから**見た目・技名・技構造・技パラメータ**を生成する。

## 3. 対象ユーザー

### 3.1 メイン対象

- オープンキャンパスに来場する高校生
- AIやゲーム制作に興味のある来場者
- 短時間で遊べる対戦ゲームを求める人

### 3.2 体験時間

1プレイあたりの目安は以下とする。

```text
キャラ入力：1〜2分
AI生成待ち：最大3分を許容（15枚並列生成のため）
技確認：30秒程度
対戦：3分以内
リザルト確認：30秒程度
```

合計で、**1組あたり4〜6分程度**の体験を目指す。

生成待ち中はトレーニングモードを提供し、待機時間の体感を短くする。

## 4. ゲーム全体フロー

### 4.1 基本フロー

```text
1. タイトル画面
2. 1P キャラクター入力
3. 2P キャラクター入力
4. AI生成中画面
5. 生成待ち中のトレーニングモード（任意）
6. キャラクター・技確認画面
7. 対戦開始
8. 試合終了
9. AIコメント生成中画面
10. リザルト画面
11. タイトルへ戻る
```

### 4.2 画面一覧

| 画面 | 役割 |
|---|---|
| タイトル画面 | 最初に表示するゲーム開始画面。豪華でスタイリッシュなビジュアルで、ゲーム開始へ誘導する |
| キャラ入力画面 | 1P/2Pがキャラ名と特徴を入力 |
| 生成中画面 | AIがキャラ・画像・技を生成していることを表示 |
| トレーニング画面 | AI生成待ち中にテストキャラで移動・ガード・回避・技の操作確認を行う |
| 技確認画面 | 生成されたキャラ画像、技名、説明、操作を確認 |
| 対戦画面 | 1vs1バトルを行う |
| AIコメント生成中画面 | 試合ログをもとにAIコメントを生成 |
| リザルト画面 | 勝者、キャラ情報、試合結果、AIコメントを表示 |

UIフォントは `keifont.ttf` を使用し、ゲームらしい太く視認性の高い表示に統一する。

キャラクター選択画面では、1P/2Pそれぞれの選択中キャラクター画像を表示し、選択変更に合わせて即時更新する。

対戦画面では、各キャラクターの頭上に `P1` / `P2` の識別ラベルを表示する。

対戦中はBGMをループ再生し、カウントダウン、試合開始、KO、メニュー操作、ジャンプ、着地、地上移動、攻撃空振り、ヒット、ガード、ガードブレイク、つかみ発生、移動回避、下回避、空中回避に効果音を付ける。近距離攻撃は空振り時に空振り音を鳴らし、ヒットしていない場合にヒット音を鳴らさない。

### 4.2 技確認画面レイアウト

AI生成完了後、対戦開始前に1P・2Pそれぞれのキャラクターを紹介する技確認画面を表示する。

**レイアウト（スマブラのキャラ紹介画面風）**

```
┌────────────────────────────────────────────────┐
│  キャラクター名（大きく中央上部）                 │
│                                                │
│  [基本技A]       ┌──────────┐  [基本技B]       │
│  技名・説明      │          │  技名・説明       │
│  右ボタン        │  キャラ  │  下ボタン        │
│                  │  立ち絵  │                  │
│  [基本技C]       │  (Idle1) │  [スマッシュ]    │
│  技名・説明      └──────────┘  技名・説明       │
│  左ボタン                       はじき＋右      │
│                                                │
│  [つかみ] LT/LB  ダメージ・ノックバックのみ表示  │
│                                                │
│         スペース / Aボタン で対戦開始           │
└────────────────────────────────────────────────┘
```

- 左側に基本技A・C、右側に基本技B・スマッシュを配置する
- 中央にキャラ立ち絵（Idle1スプライト）を表示する
- 各技はアイコン的に技名・一行説明・対応ボタンを表示する（詳細パラメーターは非表示）
- 1Pと2Pは同じ画面構成を交互に表示するか、横並びで同時表示する（実装判断に委ねる）
- 確認後、スペース/Aボタンで対戦開始

キャラクター選択画面は、保存済みキャラ・初期キャラを画面中央の**共有ロスター（スマブラ風のアイコングリッド）**にまとめて表示し、1P/2Pがそれぞれカーソルで同じ一覧から選ぶ形式とする。1Pはキャラ名（赤/青）枠で選択中セルを表す。操作は 1P=WASD・2P=矢印キー・ゲームパッドはdパッドでカーソル移動、マウスはセルの左半分クリック=1P／右半分クリック=2P で選択する。選択中キャラについては、各プレイヤーのパネルに立ち絵（待機モーション3種をループ再生）、キャラ名、技名、技の威力・射程などの主要パラメータ、地上移動速度、空中移動速度、ジャンプ力、ガード耐久値、軽さ、地上/空中回避距離を同じ画面で確認できるようにする。ロスターは3行のグリッドで表示し、1ページに収まらない場合のみページ送りを行う。カーソル移動時はグリッドを再生成せず選択カーソル色のみを更新し、待機モーション用スプライトの読み込みは選択直後フレームから外して表示遅延を抑える。ページ送りボタンはロスターの左右にアーケード調の矢印として配置し、ページ表示はヘッダー中央に置く。生成キャラの削除は誤操作防止のため確認モーダル（「削除する」／「キャンセル」、Enterで削除・Escでキャンセル）を挟む。初期キャラは削除不可。

ゲームパッドの左スティックで画面上に仮想マウスカーソル（菱形レティクル）を表示し、Aボタンでカーソル下のUI（ロスターセル・各種ボタン・トグル・確認モーダル等）を直接クリックできる。カーソルは左スティックを倒すと現れ、物理マウスを動かすと隠れる。Aがカーソルのクリックと競合しないよう、キャラ選択画面の対戦開始と削除確認の確定はキーボード（Enter/Space）専用とする。

既存キャラを使う導線と、新規キャラクターをAI生成して使う導線は分離する。既存キャラ選択画面では選択済みキャラですぐ対戦・トレーニングできる。新規生成は「キャラ生成」ボタンから専用の生成設定画面を開き、1P/2Pのキャラクター名・特徴を入力して開始する。

キャラ生成では、1Pまたは2Pの片方だけを新規生成し、もう片方は選択中の既存キャラクターを使って対戦できる。入力欄が空のプレイヤーは、AI生成や画像再生成を行わず選択中の既存キャラクターをそのまま使用する。

生成済みキャラクターはキャラクター選択画面から削除できる。初期キャラクターは削除対象にしない。

保存済みキャラクターの技JSONは、飛び道具拡張パラメータ（`projectile_angle`、`gravity_scale`、`projectile_count`、`spread_angle`、`homing`、`homing_strength`、`boomerang`）を保持する。落下弾・隕石技は保存後に再読み込みしても、上空から下向きに落ちる挙動を維持する。

## 5. キャラクター生成仕様

### 5.1 プレイヤー入力

各プレイヤーは、試合前に以下を入力する。

| 項目 | 内容 | 必須 |
|---|---|---|
| キャラクター名 | プレイヤーが考えた名前 | 必須 |
| 特徴 | 見た目、性格、戦闘スタイル、属性、モチーフなど | 必須 |

特徴プロンプト欄は、詳細なキャラクター個性を書けるように十分な入力文字数を確保する。MVPでは300文字程度まで入力できることを目安とする。

### 5.2 入力例

```text
キャラクター名：雷神ペンギン
特徴：雷をまとったペンギンの戦士。小柄だが素早く、遠距離から雷を飛ばしつつ、近づくと高速連続攻撃をする。
```

```text
キャラクター名：紅蓮の侍ドラゴン
特徴：炎を操るドラゴンの侍。大きな刀を持ち、攻撃は遅いが一撃が重い。必殺技は炎の突進斬り。
```

```text
キャラクター名：もちもちスライム忍者
特徴：かわいいスライムの忍者。分身や煙幕で相手を惑わせ、素早く近づいて攻撃する。
```

### 5.3 AIが生成する内容

AIは、各プレイヤーの入力から以下を生成する。

| 生成項目 | 内容 |
|---|---|
| キャラクター見た目説明 | 2Dアニメ立ち絵としての外見説明（日本語） |
| 画像生成用ベースプロンプト | キャラクターの外見を英語で表現した共通ベースプロンプト |
| 技3種類＋横スマッシュ | 基本技A・B・C・横スマッシュ（attack_a/b/c/smash_side） |
| 技名 | 各技の名前 |
| 技説明 | プレイヤーに表示する説明文 |
| 技構造 | Unityが実行するアクション列 |
| 技パラメータ | ダメージ、射程、発生、後隙など |
| キャラクター主要パラメータ | 地上移動速度、空中移動速度、ジャンプ力、2段ジャンプ高さ倍率、歩き速度倍率、ガード耐久値、軽さ/重さ、地上/空中横回避距離 |
| 属性 | 技の属性 |
| 状態異常 | 技が付与する状態異常 |
| 危険度 | 技の強さ・扱いづらさを示す指標 |
| 掴みパラメータ | 掴みの射程・発生・後隙 |
| 投げパラメータ | 前投げ・後ろ投げ・上投げのダメージとノックバック |

### 5.4 キャラクター画像仕様

キャラクター画像は1キャラにつき**最大15枚**を生成する。生成は以下の2段階で行い、全スプライトの見た目一貫性を保つ。

**生成フロー:**

1. **ベース画像（1枚）を生成** — `/v1/images/generations` を呼び、`base_visual_prompt` から基本の立ち絵を1枚生成する
2. **残り14枚を並列生成** — ベース画像を入力として `/v1/images/edits` を同時に14回呼び出し、各ポーズ・エフェクトを生成する。すべてのスプライトがベース画像と同じキャラクターデザインで統一される。一部のeditがAPIエラーで失敗した場合はリトライせず、該当スプライトをIdle1で代替して対戦を続行する

#### キャラクタースプライト（11枚）

| 種別 | 用途 |
|---|---|
| 待機1〜3 (Idle1/2/3) | 待機アニメーション用の3段階ポーズ |
| ジャンプ (Jump) | 空中状態の表示 |
| やられ (Damage) | 被弾・スタン時の表示 |
| つかみ (Grab) | 掴み動作中の表示 |
| ダッシュ (Dash) | 移動中の表示 |
| 技A攻撃 (AttackA) | 基本技A発動時のモーション |
| 技B攻撃 (AttackB) | 基本技B発動時のモーション |
| 技C攻撃 (AttackC) | 基本技C発動時のモーション |
| スマッシュ攻撃 (SmashSide) | 横スマッシュ発動時のモーション |

#### 技エフェクト画像（4枚、キャラなし）

| 種別 | 内容 |
|---|---|
| エフェクトA (EffectA) | 基本技Aの技エフェクト（属性に対応した色・形状） |
| エフェクトB (EffectB) | 基本技Bの技エフェクト |
| エフェクトC (EffectC) | 基本技Cの技エフェクト |
| スマッシュエフェクト (EffectSmash) | 横スマッシュの技エフェクト |

技エフェクト・当たり判定の表示位置は技ごとにバリエーションを持たせる。飛び道具は原則としてキャラクターの頭〜胸の高さから発射し、近接技は攻撃種別に応じて足元・胴体・上段などの高さと当たり判定サイズを変える。技エフェクトを生成する場合は、技の当たり判定と見た目の位置・大きさを一致させる。

技パラメータはキャラクター特徴に応じてより大きな差を付ける。近接技のリーチ、飛び道具の射程・弾速、発生、後隙、ヒット数、当たり判定の高さは、全技が同じ傾向にならないように多様化する。

エフェクト画像はキャラクターを含まず、技の属性に基づく視覚効果のみを描画する。技の性質に応じて横長・縦長の画像生成サイズを切り替える。格闘型など、キャラクターの拳・蹴り・体そのものを攻撃判定に使う技では、個別の技エフェクト画像を生成せず、キャラクターモーションと当たり判定で表現してよい。

キャラクター画像の白背景透過は、外側の背景だけでなく、腕や脚の間など輪郭の内側に残る白背景も透過対象にする。

**投げ技（前投げ・後ろ投げ・上投げ）は画像生成を行わない。** 吹っ飛び演出はUnity側のパーティクルで対応する。

| 項目 | 内容 |
|---|---|
| 画像形式 | 2Dアニメ立ち絵（キャラ）／ゲームエフェクト（技） |
| 背景 | キャラ立ち絵・技エフェクトはクロマキー用グリーン背景を基本とし、Unity側で背景を透過する |

## 6. 技生成仕様

### 6.1 技枠

各キャラクターは以下の技・アクションを持つ。

#### AIが生成する技（4種類）

| スロットID | 入力 | 役割 | 例 |
|---|---|---|---|
| `attack_a` | 右ボタン | 基本技A。自由な性質の技 | みだれ斬り、連続パンチ、火球 |
| `attack_b` | 下ボタン | 基本技B。Aとは異なる性質 | ダッシュ攻撃、範囲技、遠距離弾 |
| `attack_c` | 左ボタン | 基本技C。状態異常・移動など個性的な技 | 煙幕、カウンター、引き寄せ |
| `smash_side` | スティックはじき＋右ボタン（B/Circle） | 横スマッシュ。高威力・高後隙の強攻撃 | 紅蓮突進、雷神一撃、ナイトメアラッシュ |

基本技A/B/Cと横スマッシュはジャンプ中でも使用できる（空中攻撃、落下継続）。着地後も後隙は継続する。

技枠は固定するが、**技名・内容・パラメータはキャラクターごとにAIが生成する**。

派生攻撃（`follow_up_actions`）を生成するかどうか、どの基本技スロットに割り当てるかは、キャラクターの特徴と技コンセプトに応じてAIが判断する。「派生」「コンボ」「連撃」「連続攻撃」「追撃」「ラッシュ」などの特徴があるキャラクターでは原則として `follow_up_actions` を使う。派生・コンボ系キャラは近接技を優先して1〜2スロットに派生を持たせ、超コンボキャラ・連撃特化キャラであれば `attack_a` / `attack_b` / `attack_c` の3スロットに派生を持たせてもよい。派生受付時間 `follow_up_window` は0.3〜1.0秒、迷う場合は0.6秒を基準とする。派生はヒット時だけでなく空振り時も、受付時間内に同じ入力を押すことで発動し、最大3回まで連続できる。派生各段は単発技とし、判定数は1個に固定する。派生各段の威力は元技より大きく低く抑え、最終段以外は相手を自分側へ弱く引き寄せてコンボがつながりやすい挙動にする。

チャージ技（`chargeable`）は主に飛び道具やビーム技を想定する。フルチャージ時間 `max_charge_time` は1.0〜3.0秒、迷う場合は1.5〜2.0秒を基準とする。横スマッシュ入力と競合しやすい基本技A（右ボタン）には原則設定せず、`attack_b` または `attack_c` の `projectile` / `beam` 系スキルを優先する。

ビーム技（`beam`）は、入力直後に攻撃判定を出さず、発生まで約0.3秒の溜め時間を持つ。溜め中はキャラクターの色変化とチャージ表示で発射前であることを示す。ビーム技は必ずその技スロットの技エフェクト画像を生成し、攻撃判定表示にもその画像を使用する。画像が未生成の場合のみ長方形フォールバックになる。横に長いビームほど縦の厚みも少し増やし、細すぎる表示にならないようにする。

各技の `actions` は空にしない。`attack_a` / `attack_b` / `attack_c` / `smash_side` は、少なくとも1つは攻撃判定・接触判定・召喚・カウンター・リフレクター・バフなど、ゲーム内で明確に効果が起きるアクションを持つ。`delay`、`dash`、`teleport`、`apply_status`、`push_enemy`、`pull_enemy` の単体技は、見た目だけで何も起きない技になりやすいため禁止し、必要な場合は攻撃判定や召喚などと組み合わせる。

召喚技（`summon`）を生成する場合、召喚するものの見た目はその技スロットの技エフェクト画像（`effect_a` / `effect_b` / `effect_c` / `effect_smash`）として生成し、バトル中の召喚エンティティ表示にもその画像を使用する。AIは `skill_name` と `description` に召喚物の名前・形状が分かる語を含め、画像生成側が召喚物を技エフェクト枠で描けるようにする。

召喚物は `direction`（`forward` / `backward` / `left` / `right` / `toward_enemy` / `away_enemy` / `stationary`）、`player_controlled`、`homing`、`knockback_direction`、`status` / `status_duration` / `chance` を指定できる。`player_controlled:true` の召喚物は召喚したプレイヤーの左右入力で操縦でき、`homing:true` の召喚物は相手を追尾する。接触時は指定されたノックバック方向と状態異常を適用する。`duration` は召喚物の寿命、`status_duration` は接触時に付与する状態異常の時間として扱う。

#### 固定アクション（AI生成なし、パラメータのみ生成）

| アクション | 入力 | 役割 |
|---|---|---|
| つかみ | LT / L2 または LB / L1 | 相手を掴む。掴み成功後に投げを選択する |
| 前投げ | つかみ中＋前入力 | 相手を前方へ投げ飛ばす |
| 後ろ投げ | つかみ中＋後ろ入力 | 相手を後方へ投げ飛ばす |
| 上投げ | つかみ中＋上入力 | 相手を上方向へ投げ飛ばす |
| 地上横回避 | 地上＋ガード保持中に左右入力 | 短時間無敵になり、左右へ移動する |
| 下回避 | 地上＋ガード保持中に下入力 | その場で短時間無敵になる |
| 空中回避 | 空中＋ガード | 方向入力があれば360度方向入力のベクトルへ移動回避し、ニュートラルでも同じ回避モーションを出す |

地上で下入力を入れながらジャンプした場合は小ジャンプになる。空中で下入力を入れた場合は急降下し、通常落下より速く落ちる。ただし、急降下はジャンプ直後には発動せず、頂点付近または落下開始後に使用可能とする。急降下速度は強すぎない範囲に抑える。

空中で技を出している間も、横方向入力による空中移動は可能とする。ただし、技・ガード・回避・ジャンプなどの他行動制限は後隙が終わるまで維持する。

つかみ・前投げ・後ろ投げ・上投げの画像生成は行わない。

### 6.2 技生成の基本方針

固定技を事前に用意するのではなく、Unity側に「技を構成する基本部品」を実装し、AIがそれらを組み合わせて技レシピを生成する。

**各技が飛び道具（projectile）か近接攻撃（melee_hitbox）かは、プレイヤーの入力テキストから推論する。**

近距離技と遠距離技は、全キャラクターに両方を必ず持たせる必要はない。遠距離主体・近距離主体・設置主体・移動攻撃主体など、プレイヤーのプロンプトから読み取れるキャラクター個性を優先して4スロットを構成する。

| 入力の傾向 | 推論の方向 |
|---|---|
| 「遠距離」「弾を飛ばす」「射撃」「砲撃」「魔法弾」など | attack_a/b/cに `projectile` を多く使用 |
| 「近距離」「接近戦」「斬る」「殴る」「格闘」など | attack_a/b/cに `melee_hitbox` や `dash + melee_hitbox` を多く使用 |
| 「罠」「設置」「陣」「フィールド」「地雷」「結界」など | `trap_hitbox` や `area_hitbox` を使用し、一定時間残る置き技を混ぜる |
| 「突進」「滑る」「飛び込む」「空中」「跳ぶ」など | `dash + melee_hitbox`、`jump_attack`、`follow_owner`付きの攻撃で移動しながら攻撃する |
| 混合・バランス型 | 近接と遠距離を組み合わせて生成 |
| 明示的な記述がない場合 | キャラのモチーフ（例：弓使い→projectile、剣士→melee）から推論 |

横スマッシュ（smash_side）は原則として近接の強打（melee_hitbox）とする。ただし「大砲」「大爆発」など明らかに遠距離の必殺系である場合は projectile も可。

**生成例（遠距離主体のキャラ）:**

```text
プレイヤー入力：
雷をまとったペンギンの戦士。遠距離から雷を飛ばしつつ、近づかれたら素早く逃げる。

AI推論 → 遠距離主体
attack_a: projectile（雷弾、主力の飛び道具）
attack_b: projectile（拡散雷弾など別パターンの飛び道具）
attack_c: dash（後方へ逃げ）＋遠距離発射
smash_side: melee_hitbox（近づいた相手への強打）
```

**生成例（近距離主体のキャラ）:**

```text
プレイヤー入力：
黒い炎をまとった素早い猫の剣士。近距離で連続攻撃する。

AI推論 → 近接主体
attack_a: melee_hitbox（連続斬り）
attack_b: dash → melee_hitbox（ダッシュ斬り）
attack_c: melee_hitbox → apply_status（スタン付与の一撃）
smash_side: dash → melee_hitbox（強力な踏み込み斬り）
```

### 6.3 技構成部品

MVPでUnity側に実装する技部品は以下とする。

| 部品ID | 内容 |
|---|---|
| `melee_hitbox` | 近距離攻撃判定を出す |
| `body_hitbox` | 拳・蹴り・体当たりなど、キャラクター自身の該当部位に追従する攻撃判定を出す。格闘型・武器や魔法を使わない技では原則として技エフェクト画像を不要にする |
| `projectile` | 飛び道具を発射する |
| `area_hitbox` | 円形・矩形などの範囲攻撃を出す |
| `trap_hitbox` | 前方・足元・空中などに一定時間残る置き攻撃を出す。設置技は技判定だけが2〜5秒程度残り、使用者は後隙が終わったら移動・行動できる |
| `dash` | 前方または後方に素早く移動する |
| `jump_attack` | 空中または上方向に攻撃する |
| `counter` | 一定時間、相手の攻撃に反応して反撃する |
| `buff_self` | 一定時間、自分を強化する。プロンプトに応じて移動速度、ジャンプ力、透明化、攻撃力強化などを使い分ける。透明化は見た目のみで無敵にはしない |
| `teleport` | 短距離ワープで位置を変える補助技。攻撃判定や技エフェクトを必ず伴う必要はない |
| `pull_enemy` | 相手を自分側または中心へ引き寄せる |
| `push_enemy` | 相手を押し出す、吹き飛ばす |
| `apply_status` | 状態異常を付与する |
| `delay` | 次のアクションまで時間差を作る |
| `multi_hit` | 複数回ヒットする攻撃にする |

### 6.4 状態異常

MVPで実装する状態異常は以下とする。

| 状態異常ID | 内容 |
|---|---|
| `stun` | 0.4〜0.7秒怯ませる。操作不能時間はこの範囲に制限する |
| `burn` | 数秒間、継続ダメージを与える |
| `slow` | 一定時間、移動速度を下げる |
| `guard_break` | ガード状態を崩し、短時間無防備にする |

状態異常・デバフ中は、プレイヤーが効果を判別しやすいようにキャラクター付近へ `SLOW`、`BURN`、`STUN` などの短いラベルを表示する。

`BURN` の継続ダメージは、各ダメージ発生ごとに数値ポップアップを表示する。

### 6.5 属性

技には属性を持たせる。属性は主に演出と軽い効果に使用する。

| 属性ID | 内容 |
|---|---|
| `none` | 無属性 |
| `physical` | 物理 |
| `fire` | 炎。burnと相性が良い |
| `ice` | 氷。slowと相性が良い |
| `lightning` | 雷。stunと相性が良い |
| `dark` | 闇。pullやguard_breakと相性が良い |
| `wind` | 風。pushやdashと相性が良い |

## 7. 技パラメータ仕様

### 7.1 共通パラメータ

各技は、最低限以下のパラメータを持つ。

技の使用制限は**後隙（recovery）**が主体となる。技を出すと `startup → active_time → recovery` の順にフェーズが進み、recovery が終わるまで次の行動（技・移動・ガード）はできない。recovery の長さがそのままリスクになる。別途クールダウンタイマーは持たない。

技連打が強くなりすぎない範囲で後隙を補正する。ただし操作感が重くなりすぎないよう、過剰な後隙加算は避ける。特に、設置技・移動攻撃・状態異常付き攻撃は、当てた後の有利が大きいため同威力の単純攻撃より少し長い recovery を持たせる。

技によるガード耐久値の削りは控えめにする。通常技はガード削りを小さくし、横スマッシュでもガード耐久値を一気に大きく削りすぎないようUnity側で補正する。

| パラメータ | 内容 |
|---|---|
| `damage` | 基本ダメージ |
| `range` | 射程 |
| `startup` | 発生までの時間（この間は攻撃判定なし） |
| `active_time` | 攻撃判定の持続時間 |
| `recovery` | **後隙**。技後の硬直時間。これが終われば即座に行動可能 |
| `hit_count` | ヒット回数 |
| `knockback` | 吹き飛ばし量 |
| `stun_time` | 怯み時間 |
| `guard_damage` | ガード時に削る量 |
| `move_force` | 技による移動量 |
| `element` | 属性 |
| `status_effect` | 付与する状態異常 |
| `risk_level` | 技の危険度 |

### 7.2 HP・ダメージ基準

MVPでは以下を基準とする。

| 項目 | 値 |
|---|---|
| HP | 300 |
| 基本技A/B/Cダメージ | 6〜14（技の性質による） |
| 横スマッシュダメージ | 18〜30 |
| 掴みダメージ | なし（投げへ移行） |
| 前投げ・後ろ投げ・上投げダメージ | 8〜12＋ノックバック |
| 決着目安 | 8〜12回程度の有効ヒットで決着 |

### 7.3 HP表示

- HPバーは現在HPに応じて横方向に減少する
- HP減少時はメインバーを即時に減らし、ダメージ残量を遅れて追従させる
- HP残量に応じてバー色を緑系から黄、赤へ変化させる
- 低HP時は危険状態が分かる発光演出を入れる

### 7.4 技ごとの目安

| 技枠 | ダメージ | startup | 後隙 (recovery) | 特徴 |
|---|---:|---:|---:|---|
| 基本技A (attack_a) | 6〜14 | 0.06〜0.15秒 | 0.3〜0.8秒 | 主力技。使いやすいが後隙は短め |
| 基本技B (attack_b) | 6〜14 | 0.10〜0.25秒 | 0.5〜1.2秒 | Aと異なる性質。やや後隙あり |
| 基本技C (attack_c) | 4〜12 | 0.10〜0.30秒 | 1.0〜2.5秒 | 状態異常・移動など。後隙が長め |
| 横スマッシュ (smash_side) | 18〜30 | 0.08〜0.32秒 | 0.12〜0.55秒 | 高威力。後隙は短めだが発生とリーチでリスクを調整 |
| 前投げ / 後ろ投げ / 上投げ | 8〜12 | — | つかみ後隙に依存 | ガード貫通。ノックバック大 |

### 7.5 MP・ゲージ

MVPでは、**MPやゲージ消費は実装しない**。

技の使用制限は、主に以下で管理する。クールダウンタイマーは持たず、**後隙（recovery）が主要な制限手段**となる。

- **後隙**（recovery）: 技後の行動不能時間。これが主な再使用制限
- 発生速度（startup）
- 射程
- リスクレベル

## 8. AI生成結果の補正仕様

### 8.1 基本方針

プレイヤーは強そうなキャラクターを自由に作ってよい。  
ただし、AIやUnity側は必ずバランス補正を行う。

例：

```text
入力：全てを一撃で破壊する無敵の神

補正後：
高火力の必殺技を持つが、発生が遅く、後隙が非常に長いキャラクター
```

### 8.2 強すぎるプロンプトの扱い

以下の方針とする。

- 強そうなプロンプトは歓迎する
- ただし、ゲーム内では必ずバランス補正する
- 強すぎる表現がある場合、注意表示を出す
- 「即死」「必中」「完全無敵」などは、そのまま実装せず、リスクや制限付き効果に変換する

### 8.3 注意表示例

```text
このキャラクターは非常に強力な特徴を持っています。
ゲームバランスを保つため、技には大きな後隙が付与されます。
```

### 8.4 補正ルール

Unity側はAI出力をそのまま採用せず、以下の制限をかける。

| 項目 | 制限 |
|---|---|
| 最大ダメージ | 基本技A/B/Cは14、基本技Cは12、横スマッシュは30を目安に制限 |
| 最小後隙 | 技枠ごとに下限を設定（連打防止） |
| 最大射程 | ステージ幅に対して過剰にならないよう制限 |
| 遠距離攻撃 | projectile主体の技は遠距離から安全に振れるため、総ダメージを近距離技より明確に低く補正する。基本技の遠距離攻撃は単発でも4〜6ダメージ程度を目安にする |
| ノックバック | 技ハメを防ぐため、全体的にやや強めに補正し、命中後に相手との距離が少し離れるようにする |
| 最大怯み時間 | 長時間の操作不能を禁止 |
| 無敵 | MVPでは原則禁止。必要な場合は短時間かつ高リスク扱い |
| ガード不能 | 原則禁止。guard_breakとして短時間効果に変換 |
| 多段ヒット | 総ダメージが上限を超えないよう各ヒットを調整し、ヒット回数が多いほど1回あたりの威力を低くする。1回の技発動で複数段ヒットする場合、全段ヒット時の総ダメージは最大6程度を目安にする |

攻撃力バフ中に設置した `trap_hitbox` や飛び道具は、設置・発射時点の攻撃倍率を保持し、後から命中してもバフ効果が反映される。

### 8.5 リスクレベル

技には危険度を持たせる。

| リスク | 内容 |
|---|---|
| `low` | 扱いやすいが威力は低め |
| `medium` | 標準的な性能 |
| `high` | 強いが発生・後隙が大きい |
| `extreme` | 非常に強いが外すと大きな不利を背負う |

## 9. バトル仕様

### 9.1 対戦形式

| 項目 | 内容 |
|---|---|
| 対戦形式 | プレイヤー vs プレイヤー |
| 対戦方式 | ローカル対戦 |
| 勝利条件 | 相手のHPを0にする |
| 時間制限 | 3分 |
| ラウンド | 1本勝負（デフォルト）または2本先取（ロビーで切り替え可能） |
| タイムアップ | 残りHPが多い方の勝利 |
| ステージ | 地続きの横長ステージ、左右に中段プラットフォーム（ロビーでON/OFF切り替え可能） |
| 場外撃墜 | なし |
| 画面端 | あり |

画面端はキャラクターが画面外へ出ない範囲で、左右に十分な移動・ノックバック余地を確保する。試合終了後はスペースキーまたはエンターキーでキャラクター選択画面へ戻れる。

### 9.2 キャラクター基本アクション

| アクション | 入力 | 内容 |
|---|---|---|
| 左右移動 | 左スティック / 十字キー | 地上を左右に移動する。左スティック浅倒しは歩き、深倒し・十字キーはダッシュ |
| ジャンプ | Y / Triangle（上ボタン） | 上方向にジャンプする（2段ジャンプ可） |
| ガード | RT / R2 または RB / R1 | 前方からのダメージを軽減する |
| 回避 | 地上はRT/RB保持中に方向入力、空中はRT/RB | 地上横回避、下回避、空中回避。回避中は技が当たらない |
| 基本技A | 右ボタン（B / Circle） | 地上・空中で使用可 |
| 基本技B | 下ボタン（A / Cross） | 地上・空中で使用可 |
| 基本技C | 左ボタン（X / Square） | 地上・空中で使用可 |
| 横スマッシュ | スティックはじき＋右ボタン（B/Circle） | 地上・空中で使用可。高威力・高後隙 |
| つかみ | LT / L2 または LB / L1 | 相手を掴む。ガード貫通 |
| 前投げ | つかみ中＋前入力 | 前方へ投げ飛ばす |
| 後ろ投げ | つかみ中＋後ろ入力 | 後方へ投げ飛ばす |
| 上投げ | つかみ中＋上入力 | 上方向へ投げ飛ばす |

**ジャンプ**: 地上ジャンプ後、着地するまでに空中ジャンプを1回だけ使える（2段ジャンプ）。2段ジャンプの高さは地上ジャンプの0.3〜0.6倍程度に抑える。

**移動**: 歩きとダッシュを区別する。歩き速度はダッシュ速度の0.2〜0.5倍程度とし、アナログ入力の浅倒しでは歩き、深倒し・十字キー・キーボード左右入力ではダッシュとして扱う。

**空中攻撃**: 基本技A/B/Cと横スマッシュはジャンプ・落下中でも使用できる。使用後も落下は継続し、着地まで空中慣性が維持される。着地しても技の後隙はキャンセルされず、後隙終了まで次の行動はできない。

**キャラクター主要パラメータ**: AIはキャラクター特徴から地上移動速度、空中移動速度、ジャンプ力、2段ジャンプ高さ倍率、歩き速度倍率、ガード耐久値、軽さ/重さ、地上/空中横回避距離を推論する。軽いキャラは素早く高く跳べるがノックバックを受けやすく、重いキャラは移動・ジャンプが控えめだがノックバックを受けにくい。
移動速度・空中移動速度・ジャンプ力・軽さ/重さは、キャラ特徴がプレイ感にはっきり出るように広めの範囲で生成・補正する。

**回避**: 地上では、ガードボタンを保持した状態から左右または下入力が入った瞬間に回避を出す。左右入力なら地上横回避、下入力なら下回避になる。地上横回避と空中回避中は相手キャラクターの当たり判定をすり抜けられる。空中では、ガード入力で空中回避を出し、アナログ入力があれば360度方向入力のベクトルへ移動回避する。空中ニュートラル回避も空中横回避と同じDashスプライトのモーションを使うが、移動速度や重力を変えず無敵状態だけを付与する。空中方向回避中は最初に重力を切り、入力方向へしっかり移動する。空中回避はジャンプしてから着地するまで1回だけ発動できる。空中回避中に地上へ着地した場合、回避の後隙は即座に解除される。ジャンプと空中方向回避を高速で交互に使うことで、着地を挟んだ高速移動（絶）ができる。回避モーション中は攻撃・飛び道具・状態異常を受けず、ガードの青色流用や半透明化で回避中であることを表示する。地上横回避距離と空中横回避距離はAIがキャラクター特徴から推論し、Unity側で範囲補正する。

**ガード**: 通常ガードは地上でのみ発動できる。空中ではガードボタンを通常ガードには使わず、空中回避の入力として扱う。ガード中は移動・技を使用できない。地上ガードボタンを押している状態からは、ジャンプ入力またはつかみ入力だけがガードを解除してその動作へ移行できる。基本技・スマッシュなどの攻撃はガードボタンを離した状態からのみ発動できる。ガードで技を防いだ場合、HPダメージは無効化し、ガード耐久値のみ減少する。ガード耐久値は従来の2倍を基準とする。ガードには耐久値を設け、ガード中の時間経過および被弾時のガード削りで減少する。ガード展開中の自然減少速度は従来の0.5倍にする。ガードを離した後の耐久値回復ディレイは0秒とし、回復速度は標準値の0.8倍程度にする。耐久値が0になった場合はガードブレイクとなり、5秒間または次に技を受けるまで行動不能になる。ガードブレイク解除時は耐久値を最大まで回復する。ガードブレイク中は見た目でも判別できる演出を表示する。

**つかみ**: ガード中の相手にも当たる。掴み成功後、前後入力で投げ方向を選択する。掴み成立時に左右入力を押しっぱなしにしていた場合は即投げに移行せず、一度ニュートラルに戻してから再入力した方向で投げる。掴みに失敗（空振り）した場合は短い後隙が生じる。投げ成功後は短い再使用待ち時間を設け、連続してつかみ・投げを繰り返せないようにする。

**向き**: キャラクターは相手位置による自動振り向きを行わない。移動入力で向きが変わる。左向きで技を出した場合、技エフェクトも左右反転して表示する。

### 9.3 ノックバック

- 攻撃命中時、相手を一定距離押し出す
- 横スマッシュや投げはノックバックが大きい
- 画面端では、ノックバックによって壁に押し付けられる
- 壁バウンドはMVPでは必須ではない

### 9.6 掴み・投げ仕様

| 項目 | 内容 |
|---|---|
| 掴み射程 | AIが生成（約1.5程度） |
| 掴み発生 | AIが生成（約0.04〜0.12秒） |
| 掴み後隙 | 空振り時のみ後隙あり（成功時は後隙なし） |
| 掴みの後隙（recovery） | AIが生成（空振り時：約0.08〜0.22秒） |
| 投げダメージ | AIが生成（8〜12程度、前投げ・後ろ投げ・上投げ） |
| 投げノックバック | AIが生成。Unity側で強めに補正し、前後で方向が変わり、投げた方向へ大きく吹き飛ばす |
| つかみのガード判定 | **ガード貫通**（ガード中の相手にも当たる） |
| 投げのガード判定 | ガード貫通（つかみ→投げは一連の流れのため同様） |

**つかまれ中の挙動:**
- つかまれた側のキャラクターは**完全に操作不能**になる
- つかみが成功してから**最大1.1秒程度**で自動的に掴みが解除される（掴んだ側は投げを選択しなかった扱い。短い後隙が生じる）
- 掴み中は両キャラクターとも移動しない（掴んだ側はその場で固定）
- 掴み成立前から押しっぱなしの左右入力では投げない。掴み成立後に左右入力を一度離し、再入力した時だけ前投げ・後ろ投げへ移行する
- 投げ成功後、掴んだ側には短い再使用待ち時間が入り、すぐに再度つかむことはできない
- 掴んでいる間および掴みの後隙中、掴んだ側は Grab スプライトを維持する
- つかまれている間、相手は Damage スプライトを表示する
- 投げ成功時は相手を投げ方向へずらしてからノックバックを適用し、掴んだ側の当たり判定に引っかからず飛ぶようにする
- 上投げはつかみ中の上入力で発動し、相手を上方向へ投げ飛ばす

### 9.4 ガード

| 項目 | 内容 |
|---|---|
| ガード方式 | ボタン式ガード |
| ガード効果 | HPダメージ無効化 |
| ガード削り | 技によって少量のガードダメージあり |
| ガード崩し | 状態異常 `guard_break` により短時間ガード不能 |
| つかみへの対応 | **ガードはつかみを防げない**（つかみはガード貫通） |

### 9.5 画面端

- ステージ左右端に壁を配置する
- プレイヤーは画面外に出ない
- 画面端では逃げ場が少なくなるため、ノックバックやガード崩しが重要になる

### 9.7 中段プラットフォーム

ステージ左右に半透明の中段プラットフォームを2つ配置する。ロビー画面の「台 ON/OFF」トグルで有効・無効を切り替えられる。

- プラットフォームはワンウェイ（上から乗れる、下からはすり抜ける）
- 台の上から下入力（単独またはジャンプと同時）でプラットフォームをすり抜けて降りられる
- 台の上での向き転換・技使用は地上と同じ挙動
- 台の上でも `IsGrounded` として扱い、地上ジャンプ・技後隙・ダッシュなど地上アクションが使用可能

### 9.8 飛び道具相殺・召喚物

**飛び道具相殺（相殺システム）**

- 相手プレイヤーの飛び道具と自分の飛び道具が接触すると、双方が消滅する（相殺）
- ブーメランを含むすべての飛び道具種別が相殺対象

**召喚物の耐久**

- 召喚技で生成したエンティティはHP10を持つ
- 相手の攻撃（近接・飛び道具）が当たるとダメージを受け、HP0で破壊される
- 破壊時はダメージポップアップと「破壊!」テキストを表示する

### 9.9 KO演出

KO（HP0）時に以下の演出を行う。

- KO発生から約2.5秒間スローモーション（時間スケール大幅縮小）
- スロー中にカメラがKOされたファイターへズームイン
- ズームはロビーに戻るまで維持する（ラウンド間リセット・ロビー戻りでズーム解除）
- KO後約2.9秒（実時間）遅れてリザルトオーバーレイを表示

## 11. 演出仕様

### 11.1 立ち絵の動かし方

キャラクターは状態に応じて対応するスプライトに切り替える。スプライトの切り替えに加えて、位置・傾き・拡大縮小などのアニメーション演出を組み合わせる。

| 状態 | 使用スプライト | 追加演出 |
|---|---|---|
| 待機 | Idle1→Idle2→Idle3を約0.3秒間隔でループ | 躍動感のある連続切り替え |
| 移動 | Dash | 傾き |
| ジャンプ・落下 | Jump | 上昇・落下に合わせた傾き |
| 基本技A発動 | AttackA ＋ EffectA | 踏み込み、残像 |
| 基本技B発動 | AttackB ＋ EffectB | 踏み込み、残像 |
| 基本技C発動 | AttackC ＋ EffectC | 踏み込み、残像 |
| スマッシュチャージ中 | SmashSide | チャージゲージをキャラ頭上に表示、エフェクトが徐々に大きくなる |
| 横スマッシュ発動 | SmashSide ＋ EffectSmash | カメラ揺れ、スローモーション風演出（最大チャージ時は演出強め） |
| 後隙中 | 発動した技のスプライトを維持 | キャラが硬直している様子をそのまま表現。専用UIは出さない |
| 被弾 | Damage | 画面揺れ、点滅、後退 |
| つかまれ中 | Damage（やられモーションを流用） | 掴み側キャラの位置に固定表示 |
| ガード | （通常スプライト） | 防御エフェクト表示 |
| 回避 | Dash＋青色/半透明 | 地上横回避・下回避・空中回避中は攻撃を受けない |
| つかみ（掴む側） | Grab | 掴み開始、掴み保持中、空振り・自動解除後の後隙中も表示を維持 |

**待機アニメーション**: Idle1→Idle2→Idle3を約0.3秒ごとに切り替え、1→2→3→1→2→3…とループする。切り替えにはクロスフェードは行わず即時切り替えとし、パラパラ漫画のような躍動感を出す。

生成できていないスプライト種別がある場合は、Idle1を代替スプライトとして使用し、対戦はそのまま続行する。リトライは行わない。

キャラクター確認画面（技確認画面）では、中央にキャラ立ち絵（Idle1スプライト）、左右に各技のスキル名・説明・操作ボタンを表示する。1Pと2Pを順番に表示し、確認後スペース/Aボタンで対戦開始できる。

### 11.2 エフェクト

技の見た目は、AIの属性や技構造に応じてUnity側の汎用エフェクトを組み合わせる。

| 属性 | エフェクト例 |
|---|---|
| fire | 炎、赤系、爆発 |
| ice | 氷、青系、結晶 |
| lightning | 雷、黄色系、電撃 |
| dark | 黒紫系、影、吸い込み |
| wind | 緑系、風切り、吹き飛ばし |
| physical | 白・灰色系、衝撃波 |

### 11.3 エフェクト画像の表示方式

AI生成のエフェクト画像（EffectA/B/C/EffectSmash）は技の種類によって表示方式が異なる。

| 技の種類 | エフェクト表示方式 |
|---|---|
| 近距離・接触系（melee_hitbox, area_hitbox） | 発動時にキャラクターの前方位置に一時表示し、active_time終了後に消える |
| 飛び道具・遠距離系（projectile） | エフェクト画像を**飛び道具の当たり判定（Collider）に貼り付け**、弾と一緒に移動・消滅する |
| ダッシュ系（dash＋melee_hitbox） | ダッシュ到達後の攻撃判定位置に一時表示 |

**飛び道具エフェクトの詳細**:
- `projectile` アクションを含む技では、エフェクト画像をProjectileオブジェクトのSpriteRendererとして設定する
- 弾の移動に合わせてエフェクト画像が追従し、命中または寿命終了時に消える
- エフェクト画像のサイズは `projectile_speed` と `projectile_lifetime` から適切にスケーリングする

### 11.4 試合中AIギミック

#### 実況（CommentaryController）

## 12. AI連携仕様

### 12.1 AIを使用するタイミング

AIを使用するタイミングは以下とする。

| タイミング | 内容 |
|---|---|
| 試合前 | キャラクター見た目・技4種（attack_a/b/c/smash_side）・掴み/投げパラメータをJSON生成し、その後スプライト15枚を画像生成 |
| 試合後 | 試合ログをもとにAIコメントを生成 |

対戦中はAIに問い合わせない。  
そのため、バトル中の遅延や通信待ちは発生しない。

### 12.2 AI生成の待ち時間

試合前のみ生成するため、生成待ち時間は許容する。

生成中画面では以下を表示する。

```text
AIがキャラクターと技を生成中...
しばらくお待ちください。
```

生成待ち中は、プレイヤーが任意でトレーニングモードに入れるようにする。

トレーニングモードでは以下を満たす。

- トレーニングへ遷移してもAI生成処理は中断せず、生成完了後にキャラクター確認画面へ進める
- 生成中トレーニングから生成進行画面へ戻れる
- 生成中トレーニングでゲームパッドのBボタンを押しても意図せずメインメニューへ戻らないようにする。戻る操作は明示的なキーボードキャンセルなどに限定する
- API生成結果ではなく、事前に用意したテストキャラ・サンプル技・仮キャラ画像を使用する
- 保存済みまたは選択中キャラクターに複数スプライトがある場合、トレーニングモードでもIdle1だけでなく各モーション・技エフェクト用スプライトを使用する
- 1P/2Pの移動、ジャンプ、ガード、回避、基本技A/B/C・スマッシュ・掴みの入力確認ができる
- 時間制限と勝敗判定は行わない
- HPが0になった場合は短時間後に位置・HP・技状態をリセットする
- 生成完了またはプレイヤー操作により、キャラクター確認画面またはキャラ選択画面へ戻れる
- 生成中トレーニングから生成進行画面へ戻っても、進行中のAI生成コルーチンは停止しない

### 12.4 AIプロバイダー方針

OpenAI APIに統一する。開発・展示ともに同一プロバイダーを使用する。

#### OpenAI API（統一）

| 項目 | 内容 |
|---|---|
| テキスト生成 | OpenAI Chat Completions API（`gpt-5.4-nano`） |
| 画像生成 | OpenAI Image Generation API（`gpt-image-1`） |
| 用途（テキスト） | キャラクター見た目説明・技パラメーターJSON生成、試合後コメント生成 |
| 用途（画像） | キャラクター立ち絵生成（`visual_prompt` フィールドを渡す） |

#### API利用の設計方針

- テキスト生成（キャラ/技JSON・試合コメント）は `AICharacterClient` が Chat Completions API を呼ぶ
- 画像生成は `AIImageClient` が以下の手順で実行する
  1. `/v1/images/generations` でベース立ち絵1枚を生成（`base_visual_prompt + ", standing idle, white background"`）
  2. ベース画像を元に `/v1/images/edits` を**最大14並列**で呼び出し、残りのスプライト・必要なエフェクトを一括生成する。片方のプレイヤーだけを新規生成する場合も、そのキャラクターのベース画像生成後はベース画像以外のスプライト・エフェクトを並列生成する
  - キャラ立ち絵のeditプロンプト: `same character, {ポーズ記述子}, white background, no text`
  - 技エフェクトのeditプロンプト: `same style, 2D game visual effect, {element} type, transparent background, no character`
  - キャラクター自身に攻撃判定を付ける格闘型技や、バフ・テレポートなど攻撃判定を持たない補助技では、不要な技エフェクト画像生成をスキップする
  - 結果を `CharacterData.sprites` に格納する
  - 生成中は「ベース画像生成中...」→「スプライト生成中...」の2段階で進捗を表示する
- APIキーは環境変数 `OPENAI_API_KEY` からのみ読み込む
- 実キーはリポジトリ内ファイルに保存しない
- `YOUR_API_KEY_HERE` などのプレースホルダー値は未設定として扱い、API呼び出し前に中断する
- `AICharacterClient.ApiKey` は `AIImageClient.ApiKey` に委譲し、キー管理を一元化する

## 13. 保存仕様

### 13.1 保存対象

生成したキャラクターは、次回以降も使えるように保存する。

保存対象は以下とする。

| 項目 | 保存するか |
|---|---|
| キャラクター画像 | 保存する |
| キャラクター名 | 保存する |
| 入力された特徴プロンプト | 保存する |
| AI生成された技JSON | 保存する |
| 画像生成用プロンプト | 保存する |
| 生成日時 | 保存する |

### 13.2 保存形式

MVPではローカルファイル保存を想定する。

```text
/SaveData/Characters/{character_id}/character.json
/SaveData/Characters/{character_id}/sprites/idle1.png
/SaveData/Characters/{character_id}/sprites/idle2.png
/SaveData/Characters/{character_id}/sprites/idle3.png
/SaveData/Characters/{character_id}/sprites/jump.png
/SaveData/Characters/{character_id}/sprites/damage.png
/SaveData/Characters/{character_id}/sprites/grab.png
/SaveData/Characters/{character_id}/sprites/dash.png
/SaveData/Characters/{character_id}/sprites/attack_a.png
/SaveData/Characters/{character_id}/sprites/attack_b.png
/SaveData/Characters/{character_id}/sprites/attack_c.png
/SaveData/Characters/{character_id}/sprites/smash_side.png
/SaveData/Characters/{character_id}/sprites/effect_a.png
/SaveData/Characters/{character_id}/sprites/effect_b.png
/SaveData/Characters/{character_id}/sprites/effect_c.png
```

生成できたスプライトのみ保存し、未生成のものはスキップする。再利用時に不足しているスプライトはデフォルト画像で代替する。

### 13.3 キャラクター再利用

保存したキャラクターは、タイトル画面またはキャラ入力画面から呼び出せるようにする。

MVPでは、以下の機能を目標とする。

- 保存済みキャラ一覧を表示
- 保存済みキャラを1Pまたは2Pに読み込む
- 読み込んだキャラで再戦可能にする

AIコメント生成に失敗した場合は、Unity側のテンプレコメントを表示する。

## 15. 生成失敗時の保険仕様

オープンキャンパス展示では、通信やAPIの不具合が発生してもゲームを続行できることが重要である。

### 15.1 画像生成失敗時

画像生成に失敗した場合は、以下の順で対応する。

```text
1. 再生成ボタンを表示
2. それでも失敗した場合、事前に用意した仮キャラ画像を使用
3. キャラクター名と技はそのまま使用
```

スプライト並列生成で一部のeditが失敗した場合は、該当スプライト種別を一覧化して進捗表示に `⚠ N枚の画像生成に失敗（Idle1で代替）` と明示し、警告ログにも残す。失敗してもIdle1で代替して対戦は続行する。

### 15.2 技生成失敗時

技生成に失敗した場合は、以下の順で対応する。

```text
1. 再生成ボタンを表示
2. 再生成に失敗した場合、サンプル技セットを読み込む
3. サンプル技セットには、attack_a/b/c/smash_sideの4スロット＋掴みパラメーターを必ず含める
```

### 15.3 オフライン展示モード

APIが使えない状況に備え、事前生成済みのサンプルキャラクターを複数保存しておく。

オフライン展示モードでは、以下ができるようにする。

- サンプルキャラを選んで対戦
- 保存済みキャラを選んで対戦
- AI生成なしでデモを継続

### 15.4 API準備中の開発・展示モード

OpenAI APIキーや画像生成APIが未準備の場合でも、ゲーム本体の操作確認を進められるようにする。

- 試合前UIからトレーニングモードを開始できる
- トレーニングモードは `StreamingAssets/Presets` のサンプルキャラと `StreamingAssets/Sprites/test.jpg` をfallbackとして使用し、保存済みキャラを選んだ場合は保存済みの全スプライトセットを復元して使う
- AI生成中画面の代替として、来場者や開発者が待ち時間に操作を練習できる
- API連携が有効になった後も、生成待ち中の待機体験として残す

## 16. 安全性・表示内容の制限

オープンキャンパスで高校生が遊ぶことを想定し、不適切な入力や生成結果を避ける。

### 16.1 入力制限

以下の内容を含む入力は、警告または再入力対象とする。

- 過度に暴力的な表現
- 性的な表現
- 差別的な表現
- 実在人物への攻撃的表現
- 個人情報
- 公序良俗に反する内容

### 16.2 生成結果の制限

AI生成結果に不適切な表現が含まれる場合、以下の対応を行う。

```text
1. 表示前に検査
2. 問題がある場合は再生成
3. 再生成に失敗した場合は安全なサンプル表現に置き換え
```

## 22. 実装メモ

- キャラクター選択画面では、1P/2Pそれぞれがキャラクター名と特徴を入力できる。
- 入力が空の場合は選択中のプリセットキャラクターを使用する。
- 入力がある場合は、API連携前の暫定仕様として `PromptCharacterFactory` がローカルで対戦用 `CharacterData` を生成する。
- 暫定生成では、入力特徴から属性を簡易推定し、attack_a/b/c/smash_sideの4スロット＋掴み/投げパラメーターをサンプル技ベースで構成する。
- 画像生成が未完了の間は、選択中プリセットのキャラクター画像を流用する。
- デバッグモードはF12で開閉し、仮キャラのHP・移動速度・ジャンプ力・ガード耐久・重さを実行中に調整できる。デバッグ用チューナーはエディタ／開発ビルド時のみ自動付与し、製品ビルドには含めない。
- F12デバッグモードでは、状態異常（STUN/BURN/SLOW/GUARD BREAK）、近接、本体判定、遠距離、設置、突進、テレポート、押し出し、引き寄せ、速度/ジャンプ/透明化/攻撃バフを任意の1P/2Pで即時発動して確認できる。透明化は見た目の確認用で、無敵効果は付けない。
- 対戦中のキャラクター表示サイズ・技エフェクト・当たり判定は、直近調整値から約20%大きめを基準にする。
- CPU対戦: ロビーの「CPU」トグルをONにすると、2P側を `FighterAI` が操作する（OpenAI API非依存）。`FighterAI` は `FighterInput` と同じ `Fighter`/`SkillExecutor` の公開APIを呼ぶだけの簡易ステートマシンで、相手との距離に応じて接近・技発動・掴み・ガード/回避・牽制（飛び道具）を確率と反応遅延つきで行う。難易度は `aggression`／`defense`／`decisionInterval` で調整する。`BattleManager.ApplyCpuControl()` がトグル状態に応じて `FighterInput` と `FighterAI` を排他的に有効化する。
- F12デバッグの技発動には command_throw / barrier / shockwave / gravity_well / lifesteal / heal_self の追加アクションも含む。

## 23. 協力モード「最強ボスに挑め」（新モード・計画）

### 23.1 概要

従来の1vs1対戦に加え、**2人のプレイヤーが協力して1体の最強CPUボスに挑む**新モードを追加する。1v1 PvP・従来のCPU戦は引き続き選択可能とし、タイトル/キャラクター選択画面でモードを選べるようにする。

展示（オープンキャンパス）での「協力して強敵に立ち向かう」体験を狙いとする。

### 23.2 モード構成

- フィールドには **P1 + P2（味方）+ ボス1体**の計3体が登場する。
- 味方2人は同じ陣営。**フレンドリーファイア（味方同士の攻撃）はOFF**。
- **1人プレイ時は、2P枠を `FighterAI` の味方が自動参戦**して埋め、1人でも協力体験が成立する。AI仲間はNormal相当の強さで、ボスを狙い味方を巻き込まない。

### 23.3 ボス

- **既存プリセットキャラクターから1体を選んでボス化**する（専用ボスキャラは作らない）。
- ボスの技は、**ボス化元プリセットの生成済み技をそのまま使用し、ダメージ倍率で強化**する（専用技は作らない）。
- 強化手段は **巨大化（サイズアップ）＋ 攻撃力・速さの底上げ**。既存の `SetSizeScale` とダメージ倍率、`FighterAI.decisionInterval` 等で表現する。
- ボスのAIは**ルールベースのみ**。`FighterAI` を**複数ターゲット対応**に拡張し、2人の味方のどちらを狙うかを選択する。戦闘中のLLMは使わない。

### 23.4 勝敗・マッチ形式

- **制限時間 2〜3分**。**時間切れはボスの勝利**。
- **勝利条件**: 制限時間内にボスを撃破する。
- **敗北条件**: 時間切れ、または味方2人が同時に戦闘不能になる。
- **復活**: ダウンした味方は、**生存している仲間がダウン地点に約1.5秒留まる（踏む）と復活**する。HPは一定割合で復帰。**復活回数は無制限**（制限時間が実質の上限）。

### 23.5 アイテム（協力モードでの天使ギミック置き換え）

- 戦闘中、**一定間隔でステージのランダム位置にアイテムが出現**する。
- 効果は**味方有利の固定効果**（HP回復／各種バフ／一時無敵 等を効果表から抽選）。既存の `AngelGimmickApplier` の効果を流用する。
- 協力モードでは、従来のタイマー降臨ループ／音声入力（Whisper）／LLMによるギミック決定は**使わない**。

### 23.6 AI要素の扱い

- 「常時AI要素がある」ことは、**既存の生成AI（キャラクター生成・画像生成）で担保**する。
- ボスのLLM戦略・戦闘中のAIアシスト・AIによるアイテム効果決定は**当面実装しない**（将来の拡張候補）。

### 23.7 UI

- ボス専用の大きなHPバーを上部中央に表示し、味方2人は各自の個別HPバーを持つ。
- 3体が画面に収まるようカメラ表示を調整する。

### 23.8 段階的実装の方針（MVP）

- **MVP = Phase 1〜3**: 協力の土台（陣営/複数ターゲット/勝敗）→ ルールベースの最強ボス → アイテム取得式の支援。
- AI仲間（1人時の自動参戦）まで含めて協力体験を完成させる。
- 1v1 PvP は温存し、協力は別モードとして並走させる設計とする（既存資産への影響を最小化）。
