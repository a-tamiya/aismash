# AI Prompt Fighter 要件定義書

## 1. プロジェクト概要

本プロジェクトは、Unityで開発する2D横視点の対戦アクションゲームである。プレイヤーはテキストプロンプトを入力し、AIがその内容からキャラクター設定・技名・技の構造・技パラメータを生成する。生成された技はUnity側のスキル実行エンジンによってゲーム内で使用可能になる。

本作の中心体験は、単にAIが補助することではなく、**プレイヤーの言葉がその場でゲーム内の技として形になること**である。

---

## 2. ゲームコンセプト

### 2.1 コンセプト文

> 言葉から技を生み出し、即興で戦う2Dバトルゲーム。
> プレイヤーのプロンプトをAIが解釈し、技の構造とパラメータを生成する。
> 生成された技を使いこなし、相手を倒す。

### 2.2 仮タイトル

- Prompt Fighter
- AI Prompt Fighter
- 創造闘技場
- プロンプトファイターズ
- Skill Genesis Arena

### 2.3 企画の核

本作では、技をあらかじめ固定で用意しない。代わりに、Unity側には「技を構成するための基本アクション」を実装し、AIがそれらを組み合わせて技レシピを生成する。

例：

```text
プレイヤー入力：
黒い炎で相手を引き寄せて、最後に爆発させる技

AI生成：
範囲生成 → 引き寄せ → 遅延爆発 → ノックバック
```

この方式により、固定技のルールベース感を避けつつ、ゲームとして安全に実行できる技生成を実現する。

## 4. ゲームジャンル

### 4.1 基本ジャンル

- 2D横視点バトルアクション
- 1対1の簡易格闘ゲーム

### 4.2 操作感の方向性

本格的な格闘ゲームのような複雑なフレーム管理や長いコンボはMVPでは目指さない。まずは、生成された技を使って戦うことが楽しい、シンプルな2Dアクションを目指す。

### 4.3 目標体験

- プロンプトを書く楽しさ
- AIが技を解釈してくれる驚き
- 生成された技を実際に使える楽しさ
- 技の性能を理解して戦術を組む楽しさ
- 予想外の技が生まれる面白さ

## 6. 基本ゲームループ

### 6.1 試合前

1. プレイヤーがキャラクターのコンセプトを入力する。
2. AIがキャラクター名、設定、初期技候補を生成する。
3. プレイヤーは生成された内容を確認する。
4. 試合を開始する。

### 6.2 試合中

1. プレイヤーは通常操作で戦う。
2. プロンプトゲージが一定量たまる。
3. プレイヤーは技生成UIを開く。
4. プレイヤーが技プロンプトを入力する。
5. AIが技レシピJSONを生成する。
6. Unity側が技レシピを検証・補正する。
7. 生成技がスキルスロットに登録される。
8. プレイヤーはその技を戦闘中に使用する。

### 6.3 試合後

1. 勝敗を表示する。
2. 使用した生成技を表示する。
3. 技の評価を表示する。
4. 必要に応じて生成技を保存する。

## 8. 戦闘システム要件

### 8.1 HP

- プレイヤーと敵はHPを持つ。
- 初期HPは100を基準とする。
- HPが0以下になると敗北する。

### 8.2 ダメージ

- 通常攻撃、生成技、敵の攻撃によってダメージを受ける。
- ガード中はダメージを軽減する。
- 一部の技はガードを崩す効果を持つことができる。

### 8.3 ノックバック

- 攻撃が命中した場合、対象を一定距離押し出す。
- ノックバック量は技レシピに含める。
- Unity側で上限値を設定する。

### 8.4 クールダウン

- 生成技にはクールダウンを設定する。
- クールダウン中の技は使用できない。
- クールダウンが短すぎる場合はUnity側で補正する。

### 8.5 コスト

- 生成技にはコストを設定する。
- コストはプロンプトゲージ、MP、エネルギーなどのリソースを消費する。
- 強力な技ほど高コストになる。

### 8.6 プロンプトゲージ

- 時間経過、攻撃命中、被弾、ガード成功などで蓄積する。
- 技生成や強力な生成技の使用に消費する。
- MVPでは、技生成時に一定量のゲージを消費する。

## 9. AI技生成システム

### 9.1 基本方針

AIは、プレイヤーの入力文から以下を推論する。

- 技名
- 技のコンセプト
- 属性
- 技の構造
- アクション列
- パラメータ案
- リスク
- コスト
- クールダウン
- 演出イメージ

重要なのは、AIが単に「数値だけ」を決めるのではなく、**技の流れそのものをアクション列として生成する**ことである。

### 9.2 AIが生成するもの

AIは以下のようなJSONを返す。

```json
{
  "skill_name": "黒炎吸爆",
  "concept": "黒い炎で相手を引き寄せ、最後に爆発させる技",
  "element": "dark_fire",
  "risk_level": "high",
  "cost": 55,
  "cooldown": 8.0,
  "actions": [
    {
      "time": 0.0,
      "type": "spawn_area",
      "shape": "circle",
      "range": 4.5,
      "duration": 1.2,
      "effect": "pull",
      "power": 6.0
    },
    {
      "time": 1.1,
      "type": "damage_burst",
      "shape": "circle",
      "range": 3.0,
      "damage": 22,
      "knockback": 8.0
    }
  ],
  "vfx": {
    "color": "black_red",
    "shape": "flame",
    "impact": "explosion",
    "screen_shake": 0.35
  }
}
```

### 9.3 AIが直接行わないこと

AIは以下を直接行わない。

- Unity内のオブジェクト生成
- 当たり判定処理
- ダメージ確定処理
- プレイヤー座標の直接変更
- ゲームバランスの最終決定
- セキュリティ上危険なコード生成

AIはあくまで技レシピを生成し、Unity側がそれを検証して実行する。

## 13. バランス調整要件

### 13.1 AI出力の補正

AIが生成した数値はそのまま使わない。Unity側で必ず以下を行う。

- 最小値チェック
- 最大値チェック
- 異常値補正
- 強すぎる組み合わせの補正
- 不正なアクションタイプの除外
- 未対応属性の置換

### 13.2 基本上限値

MVPでの仮上限値：

| パラメータ | 最小 | 最大 |
|---|---:|---:|
| damage | 1 | 40 |
| range | 0.5 | 10 |
| duration | 0.1 | 5.0 |
| knockback | 0 | 15 |
| cooldown | 2.0 | 20.0 |
| cost | 10 | 100 |
| screen_shake | 0 | 1.0 |

### 13.3 パワー予算制

生成技には内部的なパワー予算を設定する。

例：

| 技カテゴリ | 予算 |
|---|---:|
| 軽量技 | 30 |
| 通常生成技 | 60 |
| 高リスク技 | 90 |
| 超必殺技 | 120 |

各効果にはコストを割り当てる。

例：

| 効果 | コスト |
|---|---:|
| ダメージ10 | 10 |
| 射程5以上 | 15 |
| 追尾 | 25 |
| スタン | 25 |
| 引き寄せ | 20 |
| 無敵 | 40 |
| ガード不能 | 35 |
| 範囲攻撃 | 20 |
| 多段ヒット | 20 |

予算を超えた場合は以下のいずれかを行う。

- ダメージを下げる
- クールダウンを伸ばす
- コストを増やす
- 発生を遅くする
- 範囲を狭くする
- 効果時間を短くする

### 13.4 危険な組み合わせ

以下の組み合わせはMVPでは禁止または強い補正対象とする。

- 無敵 + 高威力 + 短クールダウン
- 追尾 + ガード不能 + 高威力
- 広範囲 + スタン + 多段ヒット
- テレポート + 即時高火力
- 引き寄せ + 長時間拘束 + 高火力

## 14. プロンプト入力仕様

### 14.1 技生成プロンプト

プレイヤーは自然文で技を入力できる。

例：

```text
雷になって一瞬で相手の背後に回り、斬る技
```

```text
氷の鎖で相手を引き寄せて、最後に蹴り飛ばす技
```

```text
黒い炎の竜巻を作って、近づいた敵を燃やす技
```

### 14.2 入力制限

MVPでは以下の制限を設ける。

- 最大文字数：100文字程度
- 空入力は禁止
- 同一内容の連続送信は制限する
- 生成中は再入力不可

### 14.3 曖昧なプロンプトへの対応

曖昧な入力も許可する。

例：

```text
最強の技
```

この場合、AIはゲームバランス内で解釈する。ただし、曖昧な入力ほどランダム性が高く、意図通りにならない可能性をゲーム性として扱う。

## 15. AIプロンプト設計

### 15.1 システム側プロンプト方針

AIには以下を守らせる。

- 出力は必ずJSON形式にする。
- 使用可能なアクションタイプのみ使う。
- 数値は指定範囲内に収める。
- 技は2D対戦アクションで実行可能なものにする。
- 技の説明と実際のアクション列が矛盾しないようにする。
- 4つ以下のアクションで構成する。
- 過度に強すぎる技は作らない。

役割：

- プロンプトゲージを管理する

役割：

## 19. CPU要件

MVPでは高度なAI対戦相手ではなく、簡易CPUでよい。

CPUの行動：

- プレイヤーに近づく
- 一定距離で攻撃する
- たまにジャンプする
- たまにガードする
- HPが低いと距離を取る

### 19.2 将来的なCPU拡張

将来的には以下を追加する。

- プレイヤーの技使用傾向を学習
- 生成技への対策行動
- 距離管理
- ガード頻度調整
- ボスAI

## 20. データ保存要件

### 20.1 保存するデータ

MVPでは以下を保存できるようにする。

- 生成技のJSON
- 技名
- 技説明
- 使用回数
- 与ダメージ合計
- 生成元プロンプト

### 20.2 保存形式

MVPではJSONファイルとして保存する。

例：

```json
{
  "saved_skills": [
    {
      "original_prompt": "黒い炎で相手を引き寄せて爆発させる技",
      "recipe": {
        "skill_name": "黒炎吸爆",
        "element": "dark_fire",
        "cost": 70,
        "cooldown": 10
      },
      "stats": {
        "used_count": 5,
        "total_damage": 112
      }
    }
  ]
}
```

## 21. エラーハンドリング要件

### 21.1 AI生成失敗

AI生成に失敗した場合、以下を行う。

- エラーメッセージを表示する。
- ゲージは消費しない、または返却する。
- 再生成ボタンを表示する。

表示例：

```text
技生成に失敗しました。もう一度入力してください。
```

### 21.2 JSONパース失敗

JSONの解析に失敗した場合：

- 再生成を試みる。
- それでも失敗した場合はエラー表示する。

### 21.3 不正な技レシピ

不正なアクションタイプや異常値が含まれる場合：

- 該当アクションを削除する。
- 数値を範囲内に丸める。
- 実行不可なら生成失敗扱いにする。

### 21.4 通信失敗

API通信に失敗した場合：

- エラー表示
- 再試行ボタン
- 戦闘を一時停止または継続可能にする

## 22. セーフティ要件

### 22.1 入力制限

プレイヤー入力には以下の制限を設ける。

- 過度に暴力的、性的、差別的な表現をゲーム内に反映しない。
- 実在人物や既存IPに依存した生成は避ける。
- 不適切な単語を含む場合は、生成を拒否または無害化する。

### 22.2 生成結果の制限

AIが生成した名前や説明文に不適切な内容が含まれる場合、表示しない。

### 22.3 コード生成の禁止

AIにC#コードや任意スクリプトを生成させ、それを実行することは禁止する。

AIはJSONレシピのみを生成する。

## 23. 非機能要件

### 23.1 レスポンス速度

- 技生成は原則として数秒以内を目標とする。
- 生成中は専用UIを表示する。
- 生成待ち時間は「詠唱」「構築中」などの演出として扱う。

### 23.2 安定性

- AI出力が不完全でもゲームが停止しないこと。
- JSONパース失敗時にクラッシュしないこと。
- 不正な技レシピを実行しないこと。

### 23.3 拡張性

- アクションタイプを追加しやすい設計にする。
- 属性を追加しやすい設計にする。
- VFXパターンを追加しやすい設計にする。
- AIモデルを差し替えやすい設計にする。

### 23.4 保守性

- AI通信部分と戦闘処理を分離する。
- 技検証部分と技実行部分を分離する。
- UI表示とロジックを分離する。

## 25. 受け入れ基準

MVPの完成条件は以下とする。

- Unity上で2D戦闘が成立している。
- プレイヤーとCPUがHPを持ち、勝敗が決まる。
- プレイヤーが技プロンプトを入力できる。
- AIが技レシピJSONを返す。
- UnityがJSONを読み取り、技として実行できる。
- 技にはダメージ、範囲、クールダウン、コストが存在する。
- 不正なJSONや強すぎる技でゲームが壊れない。
- 少なくとも3種類以上の異なる入力から、体感の異なる技が生成される。
- 生成された技名と説明が画面に表示される。
- 生成待ち中にユーザーが状況を理解できるUIがある。

## 26. 将来的な拡張案

### 26.1 キャラクター画像生成

- プレイヤーのプロンプトから2Dキャラクター画像を生成する。

- 生成技の名前を実況に反映する。

### 26.5 ローグライト化

- 生成技を持ってステージを進む。
- 勝利後に技を強化、変化、合成できる。
- 毎回違うキャラと技で遊べる。

## 27. 現時点の設計方針まとめ

本作では、技を固定で大量に用意するのではなく、AIがプレイヤーの入力から技の構造とパラメータを推論する。

ただし、AIの出力を直接実行するのではなく、Unity側で以下を行う。

- JSONとして受け取る
- 型に変換する
- 安全性を検証する
- 数値を補正する
- 実行可能なアクション列として処理する

この設計により、以下を両立する。

- AI生成の自由度
- ゲームとしての安定性
- バランス調整のしやすさ
- Unityでの実装可能性
- プレイヤーが「自分の言葉で技を作った」と感じられる体験

---

## 28. 最重要要件

本プロジェクトで最も重要なのは、以下の体験である。

> プレイヤーが入力した言葉が、AIによって技の意味・流れ・性能に変換され、実際にゲーム内で使えること。

そのため、MVPではグラフィックやキャラクター数よりも、**プロンプトから技が生成され、使える体験**を最優先する。
